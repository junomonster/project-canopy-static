Project Canopy의 첫 번째 프로젝트는 대한민국 정부가 주도하여 공개한 범용 개발 프레임워크인 전자정부 표준프레임워크(eGovFramework)가 배포하는 공통 컴포넌트(기존 모놀리식 구조의 [common-components](https://www.egovframe.go.kr/home/sub.do?menuNo=47) 및 신규 MSA용 [msa-common-components](https://github.com/eGovFramework/egovframe-msa-common-components))를 전수 분석하는 것이었습니다. AI기반 자동화 취약점 분석 엔진을 통한 취약점 점검 진행 결과, 990건의 고유한 보안 취약점이 식별되었습니다.

해당 프레임워크가 가진 구조적 한계와 파생되는 보안 이슈에 대해서는 개발자 커뮤니티에서도 꾸준히 논의가 이루어진 바 있습니다. 실제 기업 및 기관을 대상으로 한 모의해킹 또는 보안 컨설팅에서도, 전자정부 표준프레임워크 기반 시스템에 방치된 1-day 취약점은 주요 침투 경로로 빈번하게 활용되고 있습니다.

모든 공공 웹 서비스가 해당 프레임워크를 사용하는 것은 아니며 각 서비스가 자체적으로 프레임워크를 구축하는 사례도 적지 않습니다. 하지만 정부 및 공공 프로젝트를 수주하는 대다수의 민간 SI(시스템 통합) 업체들이 이 프레임워크를 기본으로 활용하고 있어, 프레임워크의 복제된 코드는 수많은 시스템에 널리 퍼져 있습니다. 이로 인하여, 프레임워크 내부에 존재하는 하나의 취약점은 단일 사이트의 문제를 넘어, 해당 프레임워크를 사용하는 수많은 시스템의 잠재적 위협으로 직결됩니다.

따라서, 해당 프레임워크에서 발견된 990건의 보안 취약점은 매우 큰 파급력을 지닙니다. 아래는 확인된 보안 취약점의 규모와 심각도(Severity) 분포를 정리한 결과입니다.

- AI 기반 자동화 분석 엔진과 자동 Triage를 거쳐 확인된 고유 취약점은 총 990건입니다.
- 이 중 심각(Critical) 또는 높음(High) 등급에 해당하는 취약점은 94건입니다.
- 나머지는 중간(Medium)에서 낮음(Low)·정보(Info) 등급으로 분류되며, 특정 권한에서만 트리거할 수 있는 XSS 등이 여기에 포함됩니다.
- 다만 이 수치는 자동 Triage를 포함한 대량 점검 결과인 만큼, 운영 환경과 설정에 따라 실제 재현 가능성이 달라지거나 이미 인지·조치된 항목이 일부 포함되어 있을 수 있습니다.

![취약점 유형별 심각도 분포 (제보 990건)](/blog/project-canopy-egovframework-990-vulnerabilities-and-patches/overview-severity.svg)

![컴포넌트별 제보 취약점 분포 (common-components 860건 · msa-common-components 130건)](/blog/project-canopy-egovframework-990-vulnerabilities-and-patches/findings-by-component.svg)

집계된 수치는 LLM이 생성한 가공되지 않은 날 것 그대로의 결과물이 아니라, 고도화된 분석 엔진을 거쳐 오탐(False Positive) 가능성이 높은 항목을 시스템 차원에서 선제적으로 걸러낸 정제 데이터입니다. 물론 향후 전문가의 손을 거쳐 더 정확한 검증(Triage)을 거쳐야 하겠지만, 한 가지 사실만큼은 명확합니다. 최소한 취약점을 탐지하는 단계만큼은 AI 분석 기술이 압도적인 우위를 점하는 시대가 왔다는 점입니다. 사람이 수개월을 매달려도 다 보지 못할 방대한 코드에서, 오직 시스템 분석만으로 악용 가능한 수많은 보안 취약점 후보를 찾아냈습니다. 이제 진짜 과제는 다음 단계에 있습니다. "이 방대한 취약점 후보군을 누가 꼼꼼히 선별(Triage)하고, 또 어떻게 고칠 것인가"에 대한 답을 찾아야 합니다.

## 주요 취약점 사례

발견된 모든 취약점 내역을 나열하는 대신, 문제의 심각성을 보여주는 대표적인 유형들을 짚어보겠습니다(상세 분석은 아래 심층 분석 참조).

- **인증 우회(Authentication Bypass):** 로그인 처리 핸들러가 HTTP 요청 파라미터만으로 인증 상태를 부여하는 구현이 확인되었습니다. 이는 비밀번호와 2차 인증 없이 임의 이용자의 권한을 획득할 수 있는 결과로 이어집니다.
- **임의 SQL 실행을 통한 데이터 탈취:** DB 모니터링 기능에서 관리자가 저장한 SQL 쿼리를 스케줄러를 통해 그대로 실행하고 있습니다. 더욱이, 실행할 SQL 쿼리를 등록하는 엔드포인트에서 관리자 권한을 적절히 검증하지 않아, 인증된 일반 이용자도 스케줄러를 통해 임의의 SQL 구문을 실행할 수 있습니다.
- **임의 파일 탈취(Arbitrary File Access):** 첨부파일 접근을 통제하는 토큰이 프레임워크에 내장된 기본 고정 암호키를 통해 생성됩니다. 공격자는 이 키를 이용해 토큰을 위조할 수 있으며, 이를 통해 인가받지 않은 임의의 첨부파일을 다운로드할 수 있습니다.
- **안전하지 않은 직접 객체 참조(IDOR/BOLA):** 직원 정보 조회 및 수정, 첨부파일 삭제, LDAP 항목 변경 등 주요 기능에서 데이터 소유권 검증 로직이 누락되었습니다. 이로 인해 타 이용자의 민감한 자원에 무단으로 접근하고 내용을 조작할 수 있습니다. 또한, 이용자의 권한을 수정하는 엔드포인트에도 적절한 권한 검증이 존재하지 않아 관리자로 권한 상승 공격이 가능합니다.
- **출력 인코딩 부재(Stored XSS):** 공통컴포넌트의 XSS 방어는 `*.do` 요청 파라미터의 일부 특수문자만 입력 시점에 치환하는 단일 필터(HTMLTagFilter)에 사실상 전적으로 의존합니다. 이 필터는 출력 위치(컨텍스트)를 구분하지 않기 때문에, 인라인 `onclick`과 같은 HTML 속성 내 자바스크립트 컨텍스트에서는 브라우저가 속성값을 자바스크립트 해석 전에 HTML 디코딩하는 과정에서 치환됐던 문자가 원래대로 복원되어 스크립트가 실행됩니다. 또한 필터가 비활성화되거나 `*.do` 적용 범위 밖(경로 변수·외부 API·LDAP 등)으로 유입되는 입력은 필터를 거치지 않아, 출력 지점별 컨텍스트에 맞는 인코딩이 없으면 그대로 취약점으로 이어집니다.

## 표준 프레임워크 보안 취약점 패치 메커니즘의 한계

수많은 기업 및 기관의 보안 컨설팅을 수행하면서 얻은 노하우를 바탕으로 전자정부 표준프레임워크 소스 코드 저장소와 배포 구조를 분석한 결과, 반복되는 특정 문제 패턴을 확인할 수 있었습니다. 발견된 개별 취약점은 표면적인 증상일 뿐, 근본적인 원인은 코드를 작성하고 수정하며 배포하는 "과정" 자체에 있었습니다.

### 체리픽과 리베이스를 가로막는 패치 구조

전자정부 표준프레임워크의 업스트림(Upstream) 패치를 각 기관의 시스템에 적용하는 것은 기술적으로 매우 까다로운 문제입니다. eGovFrame은 외부 라이브러리처럼 의존성(Dependency) 선언 한 줄로 깔끔하게 버전을 올릴 수 있는 구조가 아닙니다. 소스 코드 전체를 복사한 후 각자의 환경에 맞게 빌드하는 템플릿 형태에 가깝습니다. 이와 같은 상황에서는 업스트림이 취약점을 패치하더라도, 일선 운영 기관에서 그 패치를 온전히 적용하기가 어렵습니다. 구체적인 사례와 이유는 아래와 같습니다.

- **식별자(CVE) 부재와 모놀리식 커밋:** 특정 취약점이 정확히 어떤 보안 위협인지 가리키는 안정적인 식별 체계(예: CVE)가 관리되지 않습니다. 또한, 보안 패치 배포 시, 실제 보안 수정 코드뿐만 아니라 대규모 리팩터링, 변수명 변경, 코드 포맷팅 등이 하나의 거대한 커밋(Monolithic Commit)으로 일괄 반영되는 경우가 존재합니다.
- **체리픽(Cherry-pick)의 불가능:** 표준 프레임워크를 사용한 개발자가 자사 시스템의 안정성을 유지하면서 "오직 해당 보안 결함만 떼어내어(Cherry-pick)" 적용하는 것이 사실상 불가능합니다. 보안 수정 사항을 파악하려면 소스 코드의 변경 단위(Hunk)를 일일이 리버스 엔지니어링해야 하기 때문입니다.
- **리베이스(Rebase) 리스크와 파편화:** 거대한 일괄 패치 전체를 적용하는 과정에서는, 필드명 변경 등으로 인한 버전 간 호환성 단절(Breaking Change)과 사이드 이펙트가 발생합니다. 결국 보안 패치 하나를 적용하는 작업이 시스템 전체를 재검수해야 하는 대형 리스크 프로젝트로 돌변하게 되고, 현장에서는 시스템 가동성을 위해 패치를 포기하는 "패치 고립" 현상이 일어납니다.

![모놀리식 보안 패치 커밋 예시](/blog/project-canopy-egovframework-990-vulnerabilities-and-patches/patch-structure-1.png)

![보안 수정과 리팩터링이 뒤섞인 diff](/blog/project-canopy-egovframework-990-vulnerabilities-and-patches/patch-structure-2.png)

결과적으로 표준 프레임워크 본체에서 취약점을 해결하더라도, 다운스트림으로의 공급망 파이프라인이 단절되어 있어 일선의 수많은 시스템에는 1-day 취약점 형태로 장기간 방치되는 악순환이 발생하게 됩니다.

### 표준이라는 신뢰가 부른 취약점 확산 리스크

공공 프로젝트에서 "표준" 프레임워크가 가지는 위상은 절대적입니다. 개발자들은 프레임워크의 기본 설정값을 "이미 검증을 마친 안전한 베이스라인"으로 신뢰하고 별도의 보안 검증 없이 아키텍처를 그대로 가져다 사용합니다.

이러한 신뢰는 역설적으로 취약점이 확산하기 매우 쉬운 환경을 조성합니다. 대표적인 예로 전역 CSRF 보호 기능이 비활성화된 `csrf=false` 설정이 기본값으로 포함되어 있지만, 이를 복사해 사용하는 일선 개발자들은 해당 디폴트 값의 위험성을 인지하기 어렵습니다. 이는 새로운 시스템을 구축할 때 기존 레거시의 예제 코드를 보일러 플레이트처럼 복사해서 활용하는 관행과 맞물려, 잠재적으로 프레임워크에 존재하는 취약한 설정 패턴이 수많은 다운스트림 시스템으로 고스란히 전파되는 위험으로 이어집니다.

보안상의 이유로 본 포스트에서 구체적인 내용을 다룰 수는 없지만, 실제로 보안 컨설팅 및 모의해킹을 수행하는 과정에서도 전자정부 프레임워크의 1-day 취약점 또는 취약한 설정을 패치하지 않고 사용하는 서비스가 다수 확인된 바 있습니다.

## 주요 취약점 심층 분석

### 임의 계정으로의 로그인

로그인 핸들러가 서버 측 검증 없이 HTTP 요청 파라미터만을 사용하여 인증 상태를 처리하는 취약점이 존재합니다. 이 결함으로 인해 공격자는 비밀번호 및 2차 인증 절차를 우회하여 임의 계정의 권한을 획득할 수 있습니다. `EgovLoginController.java` 내의 `/uat/uia/actionSecurityProcess.do` 핸들러는 안전한 서버 측 인증 로직 대신 클라이언트가 제공한 HTTP 요청 파라미터에 의존하여 인증 세션을 생성합니다. 이러한 구조적 설계 오류로 인해 공격자는 요청 데이터를 조작하여 시스템의 인증 및 잠금 메커니즘을 무력화하고, 타 사용자의 권한으로 세션을 위조할 수 있습니다.

이 취약점은 인증 프로세스를 완전히 무력화하여 비인가자가 타인의 계정을 탈취하거나 권한을 상승시킬 수 있습니다. 공격 성공시 시스템 내 데이터 무단 접근, 권한 상승 및 전체 시스템 제어권 탈취로 이어질 수 있습니다.

### CSRF를 통한 관리자 권한 획득

데이터 상태 변경(State-changing) 기능을 처리하는 컨트롤러 중 다수가 `@RequestMapping` 애너테이션 사용 시 구체적인 HTTP 메서드를 명시하지 않아, POST 요청으로 처리해야 할 중요한 작업들이 안전하지 않은 GET 요청을 통해서도 실행되는 구조적 결함이 존재합니다. 이 취약점은 특히 브라우저의 SameSite=Lax 정책을 우회할 수 있어 심각한 보안 위협을 초래합니다. SameSite=Lax 정책은 교차 사이트 요청을 기본적으로 차단하나, 사용자가 링크를 클릭해 페이지를 이동하는 최상위 GET 내비게이션 환경에서는 세션 쿠키 전송을 허용하므로, 공격자가 악의적인 링크를 노출할 경우 피해자의 브라우저는 정상 세션 쿠키를 포함한 GET 요청을 서버로 전송합니다. 실제로 분석 대상 프레임워크에서 식별된 다수의 CSRF 취약점 중 약 85%가 이와 같은 Top-level GET 내비게이션 방식을 통해 익스플로잇 가능한 것으로 나타났습니다. 공격자는 이를 통해 관리자 계정 삭제, 악성 계정 생성 등 민감한 시스템 상태 변경을 강제할 수 있으며, 결과적으로 권한 상승 및 시스템 제어권 탈취로 이어질 수 있는 치명적인 영향을 미칩니다.

CSRF 관련 결함은 공통컴포넌트(egovframe-common-components) 저장소 한 곳에서만 식별된 860건의 취약점 가운데 약 60%에 해당하는 510건이 CSRF(CWE-352)로, 단일 유형으로는 가장 큰 비중을 차지했습니다.

Project Canopy는 이를 프레임워크 수준에서 일괄 대응하기 위해, 등록, 수정, 삭제 등 상태 변경 핸들러를 POST 전용으로 제한하는 패치를 작성하여 업스트림 저장소에 제출했습니다(PR #1140). 이 패치는 112개 컨트롤러의 299개 엔드포인트에 `method = RequestMethod.POST`를 적용하고, 상태 변경을 GET 링크로 호출하던 일부 JSP 파일을 함께 수정했습니다.

## eGovFrame의 적극적인 보안 패치 반영 성과

지금까지 나열한 분석 지표들이 다소 우려스럽게 느껴질 수 있으나, 이 과정에서 반드시 주목해야 할 고무적인 대목이 있습니다. 바로 전자정부 표준프레임워크 유지보수 팀의 보안 대응 의지입니다. 이들은 외부의 취약점 제보에 대해 매우 개방적이고 적극적인 태도를 견지하며, 실질적인 결함 수정 작업을 발 빠르게 수행하고 있습니다.

실제로 Project Canopy가 1차 제보한 30건의 조치 현황을 추적한 결과, 5.0.2 패치(eb8bea2a)를 기점으로 전체의 약 80%에 달하는 23건이 최신 브랜치에서 성공적으로 패치되었습니다. 특히 관리자 계정의 무단 생성 및 삭제를 야기하던 CSRF 위협의 경우, HTTP POST 메서드 강제와 `@RequireAdmin` 애너테이션 도입을 통한 권한 검증 체계 강화로 견고하게 수정되었습니다.

아울러 KISA 및 국가정보원 등 유관 기관을 통한 정기 보안 점검 역시 체계적으로 진행되고 있습니다. 릴리스 노트를 확인해 보면, 당사가 포착한 보안 커밋 전후로도 다양한 점검 결과에 따른 조치 내역들이 꾸준히 축적되고 있음을 알 수 있습니다. 이에 더해 Pierre Kim과 같은 독립 보안 연구자들의 지속적인 제보(2025-11) 또한 프레임워크의 완성도를 높이는 자양분이 되고 있습니다.

결론적으로, 시스템이 취약해진 원인은 운영 주체의 태만이나 외부 감시의 부재에 있지 않습니다. 오히려 보안 생태계의 "책임 있는 공개(Responsible Disclosure)" 프로세스는 어느 때보다 건강하게 작동하고 있으며, 점검과 패치의 수레바퀴는 멈추지 않고 돌아가고 있습니다.

문제의 본질은 역량이나 의지의 결핍이 아닌, 물리적인 "처리량(Throughput)과 대응 속도의 격차"에 있습니다.

고도화된 자동화 분석 엔진은 단기간에 약 1,300건의 유의미한 결함 후보를 식별해 냈습니다. 하지만 이를 보안 연구자가 일일이 수동으로 재현하고 파급력을 평가해 문서화하는 방식으로는, 1차로 정식 제보까지 이어진 사례가 30건에 그쳤습니다. 방대한 후보군을 하나하나 검증하는 데 비현실적인 리소스가 필요했기 때문입니다. 제보를 받는 전자정부 메인테이너 측 사정도 다르지 않습니다. 이 정도 물량의 탐지 데이터를 검토하고 패치를 만드는 일은 그 자체로 상당한 운영 부담입니다. 결국 AI가 쏟아내는 탐지 물량을 사람 손으로 하는 수동 검증이 따라가지 못하는 "백로그 병목" 현상이, 아무리 개선 의지가 강해도 넘기 힘든 구조적 한계로 남아 있었습니다.

저희는 이 병목을 풀기 위해 사람이 직접 재현·문서화하던 과정을 자동화된 Triage 파이프라인으로 대체했습니다. AI가 도출한 1,300여 건의 후보군에 대해 재현성과 파급력을 자동으로 검증하고 우선순위를 매김으로써, 연구자의 수동 개입은 최소화하면서도 검증 품질은 그대로 유지했습니다. 그 결과 이 파이프라인을 기반으로 한 Project Canopy는 2차 제보에서 960건에 달하는 취약점을 eGovFrame 측에 추가로 전달할 수 있었습니다.

## 결론: 취약점 탐지를 넘어, 실질적인 대응 생태계 구축으로

이번 전자정부 표준프레임워크 전수 분석이 남긴 가장 큰 교훈은 발견된 취약점의 "숫자"에 있지 않습니다. 버그와 취약점을 탐지하는 기술은 이미 괄목할 만한 수준에 도달했습니다. 자동화된 분석만으로 990건의 보안 취약점을 도출해 내는 기술적 기반은 이미 마련되어 있습니다.

이제 공공 소프트웨어 생태계가 마주한 진정한 과제는 그다음 단계, 즉 "발견된 방대한 취약점 후보군을 실질적으로 검증하고 패치되게 만드는 과정"입니다. 이 영역은 단순히 우수한 자동화 기술만으로는 해결할 수 없습니다. 취약점의 타당성 검증, 패치 우선순위 선정, 책임 있는 공개 프로세스의 조율, 그리고 일선 기관(Downstream)으로의 패치 전파까지, 사람과 조직을 잇는 유기적인 협업과 자원 투입이 절대적으로 필요합니다.

Project Canopy는 이러한 난제들을 일원화된 플랫폼에서 체계적으로 해결하기 위해 결성된 공익적 협의체입니다. 본 프로젝트는 세계 최고 수준의 자동화 엔진을 기반으로 보안 취약점을 탐지하고, 이후 이어지는 정밀 선별(Triage) 및 패치 과정을 독립 보안 연구자, IT 기업, 그리고 정부 기관이 공동으로 분담하는 상호 협력적 생태계를 지향합니다. 이번 전자정부 표준프레임워크 사례에서도 Project Canopy는 취약점 탐지·제보에 그치지 않고, 검증을 마친 주요 구조적 취약점에 대한 보안 패치를 직접 작성하여 해당 오픈소스 저장소에 기여하고 있습니다. 특히 다수의 컨트롤러에 공통적으로 존재하던 CSRF 결함은 112개 컨트롤러의 299개 상태 변경 엔드포인트를 POST 전용으로 제한하는 방식으로 우선 패치하여 업스트림 저장소에 제출했으며(PR [#1140](https://github.com/eGovFramework/egovframe-common-components/pull/1140)), 객체 수준 인가(IDOR)와 관리자 트리거 XSS 대응 또한 순차적으로 추진하고 있습니다. 공공 소프트웨어의 보안 환경을 근본적으로 개선하고자 하는 본 플랫폼의 여정에 많은 기업과 연구자 여러분의 적극적인 관심과 후원을 기대합니다.
